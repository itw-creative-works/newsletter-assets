## How It Works

```
1. Cron fires            bm_cronFrequent picks up a campaign past its sendAt
2. AI generates          GPT authors content, illustrates sections as PNGs
3. Assets upload         Images + HTML + markdown archived to this repo
4. Newsletter sends      Email delivered via SendGrid / Beehiiv
```

## Architecture

Every brand gets its own branch. Zero shared history, zero cross-brand contention.

```
main                          Viewer hub (GitHub Pages)
  └── index.html              Preview any newsletter: ?path=...&branch=...

somiibo (branch)              Somiibo's newsletters
  └── content/
        ├── -OwPrc.../
        │   ├── section-1.png
        │   ├── section-2.png
        │   ├── section-3.png
        │   ├── newsletter.html
        │   ├── newsletter.md
        │   └── summary.md
        └── -OscYj.../
              └── ...

optiic (branch)               Optiic's newsletters
  └── content/...

chatsy (branch)               Chatsy's newsletters
  └── content/...
```

## Preview a Newsletter

Open the [viewer](https://itw-creative-works.github.io/newsletter-assets/) and pass a path:

```
https://itw-creative-works.github.io/newsletter-assets/?path=content/{campaignId}/newsletter.html&branch={brandId}
```

Or browse raw files directly:

```
https://raw.githubusercontent.com/itw-creative-works/newsletter-assets/{brandId}/content/{campaignId}/newsletter.html
```

## Each Issue Contains

| File | Purpose |
|------|---------|
| `section-N.png` | AI-illustrated section images embedded in the email |
| `newsletter.html` | Final rendered email-safe HTML (paste into any provider) |
| `newsletter.md` | Markdown view — one `## block` per section for block editors |
| `summary.md` | 2-3 sentence editorial recap |

## Adding a New Brand

New brands are created automatically by [Backend Manager](https://github.com/itw-creative-works/backend-manager) on first newsletter upload. The branch starts empty (orphan commit, zero files, no history from main) and accumulates issues over time.

## Safety

Everything here is **permanently public**. The upload module enforces strict allowlists — only PNGs (magic-byte verified), `newsletter.html`, `newsletter.md`, and `summary.md` are accepted. No subscriber data, API keys, or source content is ever committed.
